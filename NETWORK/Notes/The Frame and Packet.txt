
Ethernet Frame 802.3	= 14 Bytes Header + 1500 Bytes MTU (Min 20 Bytes IP Header + Min 20 Bytes of TCP Header + 1460 Bytes of TCP Segment) + 4 Bytes CRC = 1518 Bytes Frame
Ethernet Frame 802.1q	= 18 Bytes Header + 1500 Bytes MTU (Min 20 Bytes IP Header + Min 20 Bytes of TCP Header + 1460 Bytes of TCP Segment) + 4 Bytes CRC = 1522 Bytes Frame
Ethernet Frame 802.1ad	= 26 Bytes Header + 1500 Bytes MTU (Min 20 Bytes IP Header + Min 20 Bytes of TCP Header + 1460 Bytes of TCP Segment) + 4 Bytes CRC = 1526 Bytes Frame
Ethernet Preamble	= 8 Bytes
Ethernet Interframe Gap	= 12 Bytes


Ethernet Overhead		= Preamble + Frame (Header and Data) + Interframe Gap
Maximum Ethernet overhead	= Preamble (8) + Ethernet 802.1ad Frame (26) + Interframe Gap (12) = 46 Bytes

Transport/TCP

	- Header

		- Source Port

		- Sequence Number
				
			-- (Seq) indicates how many bytes have been transmitted before this packet
			— TCP Segment Length (Len) indicates the length of the TCP Segment
			- the ACK coming back will be Seq + Len

		- Acknowledgement Number	
		
			-- indicates how many bytes have been received so far

		- Header Length

		- Window Size (RWND)			
		
			-- announces the number of bytes still free in the receiver buffer
			—- in bytes, specifies the receive window of the host, each TCP segment contains the current value
			-- receiver tells sender please don’t send more than the number of bytes specified in the receive window
			—- the number of bytes the sender can still send without needing an acknowledgement from the receiver

			- 2 - MSS					-- in bytes
			- 3 - WSOPT (Window Scale)
			- 4 - TCP SACK Permitted			— len 2
			- 5 - SACK					— len N
			- 8 - TSOPT (Time Stamp)			- len 10


	- Congestion Window (CWND)				

			-- sender imposed window
			—- number of packets that can be in transit without being acknowledged, increases by 2 per ACK received,
			-- sender will not go beyond the limit of the receive window of the receiver even if congestion window allows it
			—- if sender detects packet loss it will cut the window in half


	- TCP Retransmission

			— Retransmission Timeout has initial value of 3 seconds
			- After each retransmission value of RTO is doubled
			- Maximum of 3 retransmissions (so send packet, wait 3 secs, send packet, wait 6 seconds, send packet, wait 12 seconds, give up)


	- Congestion control

		- congestion avoidance


		- fast retransmit			-- 3 ACKS received with the same number, lets retransmit from that number 


		- fast recovery

	- TCP States (example where client initiated setup and close)

		Server Side
			1. LISTEN	- waiting to receive a SYN
			3. SYN-RECEIVED	- received a SYN and sent a SYN, waiting for ACK for the SYN that was sent 
			4. ESTABLISHED	- connection established
			6. CLOSE-WAIT	- received close request (FIN) from peer, sent ACK, waiting on local app
			8. LAST-ACK	- Local app gave signal to shut, sent FIN in response, waiting for ACK to the FIN sent, received ACK

		Client Side
			2. SYN-SENT	- sent a SYN waiting for ACK
			4. ESTABLISHED	- connection established
			5. FIN-WAIT-1	- initiated FIN, waiting for ACK
			7. FIN-WAIT-2	- received ACK for initiated FIN, waiting for FIN from server, received FIN from server, sent ACK in response
			8. TIME-WAIT	- waits for time equal to double the Maximum Segment Life, timer expires
			9. CLOSED

Important Concepts

	MTU	- from interface, in bytes, this is the largest size of "IP packet" I can transmit, anything over will be fragmented,
		  if DF bit set in the packet then send back a message indicating "fragmentation needed and DF set" (type 3, code 4)

	MSS	- receive MSS is from a host, in bytes, this is the largest amount of TCP payload the host can receive		(hosts send their "receive mss" in the SYN packet)
		  send MSS is on a host, in bytes, derived from the lowest value between buffer, MTU, receive MSS and PMTUD	(hosts don’t send their "send mss")


	PMTUD	- If PMTUD is enabled on a host, and it almost always is, all TCP/IP packets from the host will have the DF bit set.
		  Only supported by TCP

	Bandwidth Delay Product		- in bytes is calculated as --> Bandwidth in Bits/8 (bits per byte) * RTT in ms/1000 (milliseconds per second)




VPN Technologies

	GRE
		- adds 24 bytes to a packet


	IPSec

		- Tunnel mode
			the entire original IP packet is protected (encrypted, authenticated, or both) and encapsulated by the IPsec headers and trailers.
			Then a new IP header is prepended to the packet, specifying the IPsec endpoints (peers) as the source and destination.

			can be used with any unicast IP traffic and must be used if IPsec is protecting traffic from hosts behind the IPsec peers

			52 bytes overhead (IPSec header, trailer and additional IP header)

		- Transport mode
			only the payload of the original IP packet is protected (encrypted, authenticated, or both). The payload is encapsulated by the IPsec headers and trailers.
			The original IP headers remain intact, except that the IP protocol field is changed to be ESP (50), and 
			the original protocol value is saved in the IPsec trailer to be restored when the packet is decrypted.

			is used only when the IP traffic to be protected is between the IPsec peers themselves

			Normally IPsec transport mode is only used when another tunnelling protocol (like GRE) is used to first encapsulate the IP data packet, 
			then IPsec is used to protect the GRE tunnel packets.
