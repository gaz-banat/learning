KERNEL SPACE vs USER SPACE (userland)

Think of each as a body of software code and memory area that the code executes in
kernel space 	-	an area in memory that only the kernel has access to - code in kernal space has direct access to the hardware, filesystem, etc.
kernel mode	-	unrestricted access to memory and cpu

user space 	-	code does not have direct access to the hardware and goes through kernel system calls
user mode	-	restricted access to a subset of memory and safe CPU operations

kernel space	-	kernel, kernel extensions, device drivers
			Kernel, IPC
			Process Scheduling
			Memory Management
			Networking Subsystem
			Virtual Files subsystem
			Netfilter
			Security Modules (SELinux modules)

user space	-	application software (programs and libraries) that interact with kernel via calls , some drivers
			Standard C library (glibc or musl or uClibc, etc)
			Init daemon (systemd)
			System Daemons (sshd, udevd)
			Window Managers (X11) 
			Graphics 
			Shells (bash)
			Utilities (iptables)

Streams stdin, stdout, stderr:
These are ‘standard I/O streams’ given to every process

These I/O streams get attached to file handles/descriptors
stdin is attached to fd 0
stdout is attached to fd 1
stderr is attached to fd 2


to know where the file descriptors for the I/O streams of a process are connected to find the process number and ls -l /proc/<process_number>/fd/
you will see where file descriptors 0, 1 and 2 are connect to.


Generally speaking for a bash process that you are in via a terminal, the process will have its I/O streams connected to that terminal, e.g. /dev/pty/X or /dev/pts/X
yep all 3 get attached to the same device file representing the terminal


These streams can be redirected to a file, a socket, another process via a pipe, etc.
stdin  		- 	‘<‘ is a way of redirecting the stdin I/O stream for a process
stdout 		-	‘>’ or ‘1>’ is a way of redirecting stdout I/O stream of a process
stderr 		-	‘2>’ is a way of redirecting stderr I/O stream of a process



What are some common /dev files
/dev/null

these 3 are a bit odd
/dev/stdin		-	a process when accessing /dev/stdin will be sent to its own fd 0 (but /dev/stdin will be given another fd number, which eventually goes back to fd 0)
/dev/stdout		-	a process when accessing /dev/stdout will be sent to its own fd 1
/dev/stderr		-	a process when accessing /dev/stderr will be sent to its own fd 2


Now look at theses 2 lines
ln -sf /dev/stdout /var/log/nginx/access.log
ln -sf /dev/stderr /var/log/nginx/error.log

The nginx process writes logs to /var/log/nginx/access.log. With the above this file is now actually just a symlink to /dev/stdout.
So what nginx now writes to its access log will come to /dev/stdout which will go the nginx process’es fd 0 (because /dev/stdout is a link to /proc/self/fd/0).
The container orchestrator will read the stdout of the nginx container process and pickup the logs 
Same for /dev/stderr and error.log

Logical Devices:

Logical devices are virtual devices that have been mapped to physical storage using the devmapper component of the kernel (dm_mod module)


E.g. /dev/dm-0 device file represents a virtual block device created by the kernel’s device mapper 
It represents a virtual hard disk that has been mapped to a physical storage device
sudo dmsetup info /dev/dm-0


E.g. /dev/mapper/sysvol device file could be a volume group on which many logical volumes are present
E.g. /dev/mapper/sysvol-root could be the logical device on which the logical volume sysvol-root is present


/dev/dm-* devices are raw devices meant for use by the kernel and can change for a device on reboot
/dev/mapper/* devices are meant for use by humans and remain persistent across reboots

NAMESPACES

This is a linux kernel feature

What:
a way to cordon off a process or processes so that they can’t reach outside of the cordoned off boundary
OR
Namespaces are a feature of the Linux kernel that partitions kernel resources such that one set of processes sees one set of resources while another set of processes sees a different set of resources


Types of namespaces:



pid		-	PID namespaces isolate the process ID number space, which means that processes in different PID namespaces can have the same PID
			A pid namespace has its own set of process ID’s starting with 1
			(don’t forget that each process on a system has a pid in its main parent namespace as well)
			
user		-	A user namespace has its own set of user IDs and group IDs for assignment to processes. 
			In particular, this means that a process can have root privilege within its user namespace without having it in other user namespaces.

net		-	Network namespaces isolate system resources that are associated with networking, namely a network stack
			Those resources include network devices, IPv4 and IPv6 addresses, protocol stacks, IP routing tables, firewall rules, socket listings, connection tracking table, 

mnt		-	the list of mount points that are seen by the processes in each namespace instance
			So a mount namespace has an indepent list of mounts as seen by processes in that namespace
			you can mount and unmount filesystems in a mount namespace without affecting the host filesystem

uts		-	Unix Time Sharing namespaces isolate two system identifiers: the hostname and the NIS domain name
			So the hostname and domain name can be different per uts namespace

ipc		-	each IPC namespace has its own IPC resources, e.g. POSIX messaging queues

cgroup

CGROUPS

This is a linux kernel feature that allows for isolating linux processes from each other

What:
Control groups, usually referred to as cgroups, are a Linux kernel feature which allow a process or a group of processes to be organized into hierarchical groups 
And Then 
 - the usage of various types of resources (cpu, memory, i/o, network) can be limited 
 - the usage of resources can be prioritized 
 - the usage of resources can be monitored/accounted.
 - the status (frozen, stopped, restarted) of all processes in a cgroup can be changed with a single command
 - also allows for a user to pin a process to a specific CPU core or a set of CPU cores

NOTE: a group of processes would be the ones running in a container

NOTE: By default, the operating system terminates a process when it exceeds a resource limit defined by its cgroup


CGroup Hierarchy (cgroup tree):
Cgroups tend to be in a hierarchy. 

Here is one hierarchy
  /user.slice

--Systemd and CGroups--

systemd heavily relies on cgroups (but systemd codebase and cgroups code base are different)

3 unit types are relevant for resource control
slice
scope
service

How:
To begin with systemd creates some top level hierarchies
- init.scope		-	in here goes the first process of systemd itself 
- system.slice		-	a top level hierarchy for processes started by root
- user.slice		-	a top level hierarchy for processes started by users
- machine.slice		-	a top level hierarchy for when this OS is running containers or vm’s

Then lets say it needs to start the sshd.service unit
It creates a cgroup /system.slice/sshd.service
The ssh process goes into that cgroup (in cgroup terminology it is a task)
Now you can set resource control on that cgroup

But the cgroup creation may not be that simple. Look at this cgroup
/user.slice/user-1000.slice/session-1.scope

DEVICES

			Kernel
			|
			|
			|
			V
Application -----> Device Driver ------> Device



There are 4 types of devices:

character		-	ls -l /dev and these will start with c, e.g. terminals
				communication is like a stream of bytes, sequential access, no random access

block			-	ls -l /dev and these will start with b, e.g. disks
				it is possible to have random access to a block in the device

pipe			-	ls -l /dev and these will start with p
				think of these as named pipes, similar to character devices, at the end of the I/O stream of a pipe is a another process and not a kernel driver

socket			-	ls -l /dev and these will start with s



network			-	generally a corresponding node file in /dev will not exist (but tun interface can have it)
				accessed by name rather than a device node file
				kernel uses a set of functions related to packet transfers to communicate with network device drivers which differ from read() and write() methods of char and block devices

Devices have numbers:

devices have a primary number and a secondary number. The primary number distinguishes between major types of devices and secondary between devices of the same type.
do ls -l /dev and you will see the primary and secondary numbers, e.g. a tty is of primary type 4, ide hard disks are of primary type 3, scsi hard disks are of primary type 8
the system selects the driver for a device based on primary number and secondary number
if you want to see the primary numbers that have been allocated for devices run `cat /proc/devices`
When the driver runs, it creates a device object for each device it finds that it can drive and assigns it a minor-device number to distinguish between different devices. So that when an application accesses a device node, the driver knows which device it is accessing based on the minor number.


How are devices discovered:

The kernel detects devices and puts them into /sys accordingly. /sys is a memory based file system
The kernel (it seems) also creates a /dev node file
The kernel sends a uevent to a netlink socket
udev (systemd-udevd) listens to the uvent on the netlink socket and loads all the attributes in the event
then udevd parses its rules, filters and updates the uevent based on those rules and then acts on the /dev node file accordingly based on rules in either /lib/udev/rules.d (defaults) or /etc/udev/rules.d (overrides)
udev creates further symbolic links within /dev for the device and initializes the device
e.g. if udev needs to create a /dev node file it will make the mknod() system call (there is also a mknod command available)



Not all devices have device files:

e.g. network devices

Interface types:

bridge			-	this interface behaves like a network switch, forwards packets between interfaces connected to it
				ip link add br0 type bridge; ip link set <interface> master br0;

bond			-	an interface that aggregates multiple network interfaces into a single logical "bonded" interface
				ip link add bond1 type bond miimon 100 mode active-backup; ip link set eth0 master bond1; ip link set eth1 master bond1;

vlan			-	creates a subinterface out of an interface and puts it into its own broadcast domain
				vlan allows the creation of multiple interfaces (eth0.2, eth0.3, etc.) on top of a single one (eth0) and filter packets based on a vlan tag
				there is only one mac address and that is of the main interface (eth0)
				ip link add link eth0 name eth0.2 type vlan id 2; ip link add link eth0 name eth0.3 type vlan id 3;

macvlan			-	allows creation of multiple interfaces, each with a different layer2 address (mac address), on top of a single interface
				there are 5 macvlan types - private, VEPA, bridge (most common, all macvlan interfaces are connected to each other via a bridge using the physical interface), passthru, source
				ip link add macvlan1 link eth0 type macvlan mode bridge; ip link add macvlan2 link eth0 type macvlan mode bridge;
				ip netns add net1; ip netns add net2; ip link set macvlan1 netns net1; ip link set macvlan2 netns net2

veth			-	virtual ethernet, this is a local ethernet tunnel, devices are created in pairs.
				packets transmitted on one device in the pair are immediately received on the other device in the pair. When either device is down the link state of the pair is down.
				use this configuration when network namespaces need to communicate to the main host namespace or between each other
				ip netns add net1; ip netns add net2;
				ip link add veth1 netns net1 type veth peer name veth2 netns net2

dummy			-	an interface that is entirely virtual, e.g. loopback. it just provides a device to route packets through without actually transmitting them.
				ip link add dummy1 type dummy; ip addr add 1.2.3.4/24 dev dummy1; ip link set dummy1 up


tun			-	a TUN interface is a software interface implemented in linux kernel
				it is an emulation of a layer 3 interface
				it can pass raw IP packets between user program (e.g. flannel) and kernel
				when program writes packets to TUN, the packet will be sent to kernel directly and kernel will route according to route table
				when packets arrive to the kernel, if route table points to TUN for the packets, kernel will send packets to the program that created the TUN interface


tap			-	emulation of a layer 2 interface similar to TUN interface

team

vxlan			-	Virtual eXtensible Local Area Network, it is a tunneling protocol established between source and destination network devices
				it works by segmenting layer 2 ethernet frames and encapsulating them in UDP packets
				so think about it as a networking protocol that tunnels layer-2 traffic over an IP network using the UDP protocol
				VTEP - VXLAN Tunnel Endpoint, VNI - VXLAN Network Identifier
